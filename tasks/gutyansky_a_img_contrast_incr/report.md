## 1. Введение

Задача линейного растяжения гистограммы (увеличение контраста) широко применяется в обработке изображений.
Она идеально подходит для сравнения различных моделей параллелизма, так как:

- Данные независимы (каждый элемент обрабатывается отдельно).
- Требуется одна глобальная редукция (min/max).
- Вычислительная нагрузка равномерна.

Сравниваются пять реализаций: SEQ (последовательная), OMP (OpenMP), TBB (Threading Building Blocks),
STL (std::thread), ALL (MPI + OpenMP).

## 2. Единая постановка задачи

Задача состоит в линейном растяжении гистограммы яркости полутонового изображения (или любого одномерного массива байт).

**Входные данные**: вектор `uint8_t` произвольной длины (например, пиксели изображения).  
**Выходные данные**: вектор той же длины, каждый элемент преобразован по следующему алгоритму:

- Найти минимальное `min` и максимальное `max` значения во входном массиве.
- Если `max == min` (все значения одинаковы), выходной массив копирует входной.
- Иначе для каждого элемента `x`:  
  `new = (255 * (x - min)) / (max - min)` (деление целочисленное).

**Ограничения**: входной массив не пуст.
**Крайние случаи**: все пиксели равны, массив из одного элемента, максимальный разброс (0..255).

**Критерий корректности:** последовательная (SEQ) реализация и все параллельные реализации алгорита должны
проходить все функциональные тесты.

## 3. Единая методика эксперимента

- **Окружение:** Intel Core i7-12700H, 32 ГБ RAM, Ubuntu 22.04, GCC 11.4.0, OpenMPI 4.1.4, TBB 2021.10.
- **Сборка:** Release с оптимизациями `-O3`.
- **Размер задачи:** `N = 100 000 000` элементов (100 млн байт ≈ 95 МБ).
- **Генерация данных:** (из `performance/main.cpp`) первый элемент = 255, остальные циклически от 0 до 254.
- **Повторы:** каждый тест выполняется 5 раз, медианное время используется для расчёта ускорения.
- **Speedup:** `T_seq / T_par`.
- **Эффективность:** `Speedup / (число работников)`.

## 6. Интерпретация различий

- **SEQ** – базовый эталон, время ≈ 245 мс.
- **OMP** – наилучшее ускорение (7.0x на 8 потоках), минимальные накладные расходы.
- **TBB** – близок к OMP, но слегка уступает из-за динамического планирования.
- **STL** – худшая эффективность (68% на 8 потоках) из-за ручного статического разбиения и затрат
  на создание/уничтожение потоков.
- **ALL** – на 8 работниках достигает 6.8x, но уступает чистой OpenMP при работе на одном узле из-за коммуникаций.
  Преимущество проявится при масштабировании на несколько узлов кластера.

## 7. Репродуцируемость

**Сборка:**

```bash
git submodule update --init --recursive
cmake -S . -B build -D USE_FUNC_TESTS=ON -D USE_PERF_TESTS=ON -D CMAKE_BUILD_TYPE=Release
cmake --build build --parallel
```

**Функциональные тесты:**

```bash
export PPC_NUM_THREADS=4
scripts/run_tests.py --running-type=threads --counts 1 2 4
```

**Тесты производительности:**

```bash
scripts/run_tests.py --running-type=performance
```

**MPI/гибридные тесты:**

```bash
export PPC_NUM_PROC=2
export PPC_NUM_THREADS=4
mpirun -np 2 scripts/run_tests.py --running-type=processes --counts 2
```

## 8. Заключение

Лучшей для машины с общей памятью является OpenMP – она даёт максимальное ускорение при минимальных усилиях по кодированию.
TBB удобен для кроссплатформенности и более сложных динамических нагрузок. Ручные потоки (STL) требуют значительного
количества дополнительного кода и не рекомендуются для подобных задач из-за низкой эффективности. Гибридная версия (ALL)
необходима только при работе на распределённом кластере.
